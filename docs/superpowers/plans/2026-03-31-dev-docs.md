### Task 2: `docs/dev/architecture.md` — 架構設計

**Files:**
- Create: `docs/dev/architecture.md`
- Read: `main.py`, `threads_client.py`, `analyzer.py`, `insights_tracker.py`, `report.py`, gstack 設計文檔

- [ ] **Step 1: 寫架構文件**

包含以下段落：

**1) 產品定位**（從 gstack 設計文檔提取）
- 自建社群經營戰情室
- 核心痛點：每天經營 Threads，但追蹤成效的方式是「打開 app 一篇篇看」
- 終極目標：全自動循環（收集趨勢 → AI 分析 → 建議發文 → 發文 → 追蹤成效 → 調整策略）

**2) Pipeline 流程圖**（Mermaid）
- cron 觸發 → `main.py` 載入 config
- Job A: `threads_client.fetch_posts` → `analyzer.analyze_posts` → `report.render_report`
- Job B: `insights_tracker.fetch_and_store_post_insights` → `fetch_and_store_account_insights` → `get_trend` / `get_top_posts` → `report.render_dashboard`
- 合併戰情日報
- 兩個 Job 獨立 try/except，互不影響

**3) 模組職責**（每個 2-3 行）
- `main.py` — 進入點、`.env` 載入、`${ENV_VAR}` config 插值、Job A + Job B 編排
- `threads_client.py` — Threads API keyword_search 封裝、跨關鍵字去重、指數退避重試 + 429 rate limit
- `analyzer.py` — `claude -p` subprocess 批次分析（每批 50 篇）、JSON 解析（含 markdown code block 處理）、fallback 機制
- `insights_tracker.py` — me/threads + insights API 抓取、SQLite upsert（post + account）、趨勢計算、Top N 查詢
- `report.py` — Jinja2 渲染日報 + 戰情日報、分類排序（新工具→產業動態→教學→觀點）、`save_report` 存檔

**4) 關鍵設計決策**

| 決策 | 選擇 | 原因 |
|------|------|------|
| AI 分析方式 | `claude -p` subprocess | 不額外接 API，用手邊有的最簡單方式；不需要管 API key、token 計費 |
| 資料儲存 | SQLite | 比 CSV 查詢能力強（趨勢計算、Top N），比外部 DB 不需額外設定 |
| Config 格式 | YAML + `${ENV_VAR}` 插值 | 可讀性好，secrets 不進版控 |
| 重複執行 | upsert（ON CONFLICT DO UPDATE） | 同一天重跑不會產生重複資料 |
| 錯誤隔離 | Job A / Job B 獨立 try/except | 趨勢收集失敗不影響成效追蹤，反之亦然 |

### Task 3: `docs/dev/data-model.md` — 資料模型

**Files:**
- Create: `docs/dev/data-model.md`
- Read: `insights_tracker.py`（schema + SQL）、`config.yaml`、`analyzer.py`（貼文結構）

- [ ] **Step 1: 寫資料模型文件**

包含以下段落：

**1) SQLite Schema**

`post_insights` 表：
- 欄位：collected_date, post_id, text_preview, posted_at, username, views, likes, replies, reposts, quotes
- PK: `(collected_date, post_id)` — 同一天同一篇只存一筆
- upsert SQL（UPSERT_POST_SQL）

`account_insights` 表：
- 欄位：collected_date, followers, total_views, total_likes, total_replies, total_reposts
- PK: `collected_date` — 每天一筆
- upsert SQL（UPSERT_ACCOUNT_SQL）

DB 位置：`data/threads.db`（由 `config.insights.data_dir` + `config.insights.db_name` 決定）

**2) 追蹤指標**

| 指標 | 計算方式 | 來源 | 用途 |
|------|----------|------|------|
| 每篇觀看數 | `views` | post insights API | 找出爆文模式 |
| 互動率 | `(likes+replies+reposts+quotes) / views * 100` | post insights | 內容品質指標 |
| 粉絲數變化 | `latest.followers - oldest.followers` | account insights | 成長趨勢 |
| 每日總觀看 | `total_views` | account insights | 整體曝光趨勢 |
| N 天趨勢 | `get_trend(conn, days=7)` | account_insights 最近 N 筆 | 趨勢方向和百分比 |

**3) config.yaml 說明**

四個區塊逐一說明：
- `threads` — access_token（`${THREADS_ACCESS_TOKEN}`）、keywords 清單、sort、max_posts_per_keyword
- `anthropic` — model、max_tokens
- `insights` — data_dir、db_name、backfill_days、timezone
- `output` — directory

`${ENV_VAR}` 插值機制：`main.load_config()` 用 regex `\$\{(\w+)\}` 替換為 `os.environ` 的值

**4) 貼文資料結構演變**

```
API 原始回傳
  → { id, text, username, timestamp, like_count, permalink }

經過 fetch_posts() 去重
  → 加上 keyword_matched

經過 analyze_posts() AI 分析
  → 加上 category（新工具/產業動態/教學/觀點）
  → 加上 score（1-5）
  → 加上 summary（繁中 20 字以內）
```

### Task 4: `docs/dev/setup.md` — 環境設定

**Files:**
- Create: `docs/dev/setup.md`
- Read: `config.yaml`、`main.py`（_load_dotenv）、gstack 設計文檔（constraints）

- [ ] **Step 1: 寫環境設定文件**

包含以下段落：

**1) 系統需求**
- Python 3.13+
- Claude Code CLI（`claude` 指令可用）
- Windows 11（開發環境）

**2) 依賴安裝**
```bash
pip install requests pyyaml jinja2
```

**3) Threads API Token 設定**
- 需要 Meta Threads API Standard Access
- 限制：只能讀自己的貼文和 insights（商家驗證中才能搜尋別人的）
- Long-lived token 有效期 60 天
- 換發流程：用舊 token 呼叫 `/access_token` endpoint 換新的
- 注意：token 過期後 pipeline 會靜默失敗（API 回 403）

**4) `.env` 設定**
```
THREADS_ACCESS_TOKEN=你的token
```
放在專案根目錄，`.gitignore` 已排除

**5) 執行方式**
```bash
PYTHONUTF8=1 python -m threads_pipeline.main
```
Windows 上需要 `PYTHONUTF8=1` 確保中文輸出正常

**6) 部署排程（cron）**
```bash
# 每天 UTC 00:00 執行（台灣時間 08:00）
0 0 * * * cd /path/to/threads_pipeline && PYTHONUTF8=1 python -m threads_pipeline.main >> logs/pipeline.log 2>&1
```

Windows 可用 Task Scheduler 替代

包含以下段落：

**1) 總覽表**

| Phase | 內容 | 狀態 |
|-------|------|------|
| V1 | 趨勢收集 | 已完成 |
| V2 | 成效追蹤 | 程式碼完成，待驗證 |
| V2.5 | 共鳴度分析 | 構想 |
| V3 | 議題分析 + 內容資產追蹤 + 分享靈感 | 構想 |
| V4 | 差異化觀點 + 全自動循環 | 願景 |

**2) V1：趨勢收集（已完成）**
- keyword_search → 去重 → `claude -p` AI 分析 → Jinja2 Markdown 日報
- 分類：新工具 / 產業動態 / 教學 / 觀點
- 評分：1-5 星趨勢分數

**3) V2：成效追蹤（程式碼完成，待驗證）**
- 抓所有貼文 + insights 數據 → SQLite
- 抓帳號層級 insights（粉絲、觀看）
- 計算 7 天趨勢、Top 5 貼文
- 合併趨勢日報 + 成效分析 → 戰情日報
- 待驗證：insights API 實際回傳格式、rate limit、backfill 歷史資料

**4) V2.5：共鳴度分析（構想）**

核心理念：「分享自己，不是推銷自己。讓讀者認識你是誰。」

擴充 analyzer 的評分維度：
- 鉤子（有沒有讓人停下來的開頭）
- 真實經驗/故事（不是空談）
- Takeaway（讀者帶走什麼）
- 聚焦度（1 篇 = 1 個核心觀點）

**5) V3：議題分析 + 內容資產追蹤 + 分享靈感（構想）**

議題分析（外部趨勢）：
- 熱門議題偵測：哪些關鍵字/主題討論量爆增
- 議題生命週期：從出現到退燒的時間軸
- 意見光譜：同議題的觀點分布

內容資產追蹤：
- 追蹤每篇貼文的長期影響力（不只看當天）
- 哪些是「常青內容」
- 哪種主題/結構的貼文生命週期最長

分享靈感（交叉分析）：
- 議題缺口：外面在討論但你還沒寫的主題
- 差異化觀點：你的觀點跟主流不同的地方
- 時機建議：某議題正在升溫，現在切入正好

**6) V4：全自動循環（願景）**

```
收集趨勢 → AI 分析 → 建議發文 → AI 草稿 → 人工確認 → 自動發布 → 追蹤成效 → 調整策略 → 循環
```

**7) 設計前提**
- 核心身份：實際在用 AI 工具做事的人，分享過程和思考
- 內容策略：不推銷，分享自己讓讀者認識你
- Side project，先自己用好再說
- 漸進式擴充：每個 phase 都是上一個的自然延伸
