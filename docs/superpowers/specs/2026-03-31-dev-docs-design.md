## 各文件內容規格

### 1. README.md — 文檔索引

入口頁，列出所有文件及一句話說明。含目錄樹。

### 2. architecture.md — 架構設計

- **產品定位**：自建社群經營戰情室，核心痛點是成效追蹤（從 gstack 設計文檔提取）
- **Pipeline 流程圖**：cron → Job A 趨勢收集 → Job B 成效追蹤 → 合併戰情日報；兩個 Job 獨立 try/except
- **模組職責**：`main.py`、`threads_client.py`、`analyzer.py`、`insights_tracker.py`、`report.py` 各 2-3 行
- **關鍵設計決策**：
  - 為什麼用 `claude -p` subprocess 而不是 Anthropic SDK
  - 為什麼選 SQLite 而不是 CSV
  - `${ENV_VAR}` config 插值機制
  - upsert 策略（同一天重跑不重複）

### 3. data-model.md — 資料模型

- **SQLite Schema**：`post_insights`（複合 PK: collected_date + post_id）、`account_insights`（PK: collected_date）、欄位定義、upsert SQL
- **追蹤指標表**：每篇觀看數、互動率、粉絲變化、每日總觀看、發文時間 vs 互動、內容分類 vs 成效
- **config.yaml 說明**：四區塊（threads / anthropic / insights / output）、各欄位用途和預設值、`${ENV_VAR}` 插值
- **貼文資料結構**：經過 pipeline 各階段後 dict 欄位的演變（API 原始 → +keyword_matched → +category/score/summary）

### 4. setup.md — 環境設定

- **依賴安裝**：Python 3.13+、pip install requests pyyaml jinja2
- **Threads API Token**：Standard Access 限制、long-lived token 60 天有效期、換發流程
- **`.env` 設定**：必要變數
- **Claude Code CLI**：需安裝才能跑 analyzer
- **部署排程**：cron 設定範例

### 5. testing.md — 測試架構

- **測試結構**：3 個測試檔案對應 3 個模組
- **Mock 策略**：threads_client mock API、analyzer mock subprocess、report 用 tempdir
- **共用 Fixture**：conftest.py 的 fixture 清單及用途
- **常用指令**：跑全部、跑單檔、跑單一測試

### 6. roadmap.md — 路線圖

| Phase | 內容 | 狀態 |
|-------|------|------|
| V1 | 趨勢收集：keyword search → AI 分析 → 日報 | 已完成 |
| V2 | 成效追蹤：insights → SQLite → 戰情日報 | 已完成 |
| V3 | 內容靈感 + AI 互動建議：該回誰的留言、建議發文主題、最佳發文時間 | 規劃中 |
| V4 | 全自動循環：AI 草稿 → 人工確認 → 自動發布 → 追蹤成效 | 願景 |

含每個 phase 的設計前提、終極目標（全自動循環：收集趨勢 → AI 分析 → 建議發文 → 發文 → 追蹤成效 → 調整策略）。
