## 候選加強點（按 value/effort 排）

### 1. Semantic Freshness Gate（小 / 高）

- **AK 來源**：`skills/draft/SKILL.md` + `skills/topics/SKILL.md` + `scripts/update_topic_freshness.py`
- **功能**：寫文/選題前比對歷史主題庫，擋掉「N 週前才寫過的角度」
- **吸收位置**：`advisor.py` review 流程 + `insights_tracker.py` 接主題標籤
- **落地路徑**：加個相似度閘門接歷史主題

### 2. AI-tone Detection（中 / 高）

- **AK 來源**：`knowledge/ai-detection.md`（句式/對比/金句密度清單）+ `skills/analyze/SKILL.md`
- **功能**：結構化 AI 味檢測清單，逐句掃
- **吸收位置**：抽成 `ai_detection.py` 模組接進 `analyzer.py`
- **優點**：現有 advisor 靠 Codex 給評語，沒有結構化硬指標

### 3. Review → Tracker 回饋迴路（大 / 最大）

- **AK 來源**：`skills/predict/SKILL.md` + `skills/review/SKILL.md`
- **功能**：`/predict` 給互動區間 → `/review` 寫回實際 → 下次校準，系統越用越準
- **吸收位置**：在 `insights_tracker.py` SQLite 時序上疊「預測 vs 實際」對比表
- **備註**：最重要的長期架構升級，但工作量最大，適合分批

## 候選點 #1 深入研究：Semantic Freshness Gate（2026-04-22）

### 校正：它其實是兩個正交系統

初次對照時把它講成一件事，細讀後校正：AK 的 "Freshness Gate" 是**兩個獨立機制**綁在一起。

**System A — 自我重複偵測（離線群聚）**
- 實作：`scripts/update_topic_freshness.py`（純 Python 無外部依賴，376 行）
- 特徵：word token + 字元 n-gram (2-3 gram) + topic 標籤 + content_type
- 相似度：加權 Jaccard（詞 0.5 + 字元 0.3 + 主題 0.2）
- 群聚：Union-Find，threshold 0.24
- 新鮮度分數衰減：
  - 相似近期貼數 × 18（最多 -54）
  - 30 天內同集群頻率 × 8（最多 -24）
  - 距上次同集群天數：3d=-25 / 7d=-18 / 14d=-10 / 30d=-5
- 結果寫進 `post.algorithm_signals.topic_freshness.*`

**System B — 外部飽和度檢查（即時 WebSearch）**
- 位置：`/draft` Step 2.5 + `/topics` Step 3.5
- 邏輯：WebSearch 關鍵字 → 分類 Green / Yellow / Red
- 失敗封閉：WebSearch 不可用時不得偷標 Green
- 審計 log：`threads_freshness.log` JSONL 每次執行寫一行

**兩者正交**：A 回答「我最近自己有沒有寫過類似的」；B 回答「世界上是不是寫爛了」。catch 不同的 failure mode。

### 我們現況的精確 gap

檢查 `advisor.py:97-102`：
```python
if days_since > 3:
    freshness = "已過期，建議先暖場再發文"
elif days_since > 1:
    freshness = "冷卻中，可以發文"
else:
    freshness = "仍在 freshness 週期內"
```

這個 `freshness_status` 只是**距上次發文的天數**，是帳號層冷卻狀態，**不是語意新鮮度**。

真正跟 AK System A 對應的缺口：
- 無語意群聚，雖然 SQLite 的 `full_text` 欄位原料齊全
- 無結構化「自我重複」訊號傳給 Codex review
- Review prompt 的「定位一致性」維度靠 Codex 主觀判斷，沒有量化指標

### 關鍵設計決定

| 決定項 | 結論 | 備註 |
|-------|-----|-----|
| **Q1 運算時機** | **方案 C（簡化即時版）** | 每次 `advisor review` 時，SQL 拉最近 N 天貼文，Python 算 draft vs 近期，不做全史群聚、不寫回 DB |
| **Q2 外部飽和度（System B）** | **暫緩** | 等 Threads Advanced Access 核准或另有 WebSearch 通道再評估 |
| **Q3 Jaccard + n-gram 對繁中的誤群風險** | **動工前必測** | 擔心：繁中 AI/tech 貼文常含「Claude」「AI」「Agent」等高頻詞，可能被跨主題誤群。驗證方式：把現有貼文餵進 AK script 跑一次看分群結果 |
| **Q4 Gate 強度** | **打分數不硬擋** | 輸出分數讓使用者自己判斷；不做 Red → 禁止寫 |

### Q1 方案 C 的細節（採用的版本）

**邏輯流程**：
```
SQLite (source of truth，資料不動)
    ↓ SELECT full_text, posted_at FROM post_insights WHERE posted_at > NOW() - 30 days
Python 函式（borrow AK 的 extract_word_tokens + extract_char_ngrams + semantic_similarity）
    ↓ draft vs 每篇歷史逐一比對
輸出：max_similarity + 最相似的一篇 + N 天前發的
    ↓
advisor review prompt 帶入這個結構化訊號
    ↓
Codex 在「定位一致性」維度用這個量化資料判斷（而非憑感覺）
```

**特性**：
- 純函式，~50-100 行可完成
- 不寫回 SQLite、不新增欄位、不做 migration
- 拆掉只需刪函式，不影響既有資料

### 為什麼拒絕的方案

**方案 A（批次群聚 + 快取）**：貼文量 < 100 時 cluster 多半只 1-2 篇，群聚意義小；要加新欄位+migration，與收益不成比例。等貼文量到 100+ 再升級。

**方案 B（純即時全量群聚）**：每次 review 都重跑 O(n²)，貼文多了會慢。沒必要為了全局觀而每次重算。

**SQLite FTS5**：中文斷詞需 ICU tokenizer，Windows 設定麻煩；BM25 是 query→doc 設計，硬拿來做 doc→doc 要繞；跟 AK 的 Jaccard 結果意義不同無法對照。設定成本 >> 收益。

**Vector embedding（sqlite-vec 等）**：最強但要對每篇呼叫 embedding API，多一層外部依賴；對 < 100 篇規模是殺雞用牛刀。未來值得考慮，現在不划算。

### 不 fork AK 腳本的原因

只 borrow 他的邏輯：
- `extract_word_tokens`
- `extract_char_ngrams`
- `is_non_ascii_letter`
- `semantic_similarity`（加權 Jaccard）

這四個是純函式、算法明確、好抽取。其他（Union-Find 群聚、tracker.json I/O、CLI wrapper）我們不需要。

好處：保持資料主權（SQLite），不依賴他的 JSON 格式，他未來的更新不用追。

### Q3 驗證風險的具體方法（動工前要做）

在寫新程式前，先用 AK 原 script 驗證一次誤群風險：
1. 把 SQLite 貼文匯出成符合 AK `tracker.json` 格式的 minimal JSON
2. 跑 `python .reference/AK-Threads-booster/scripts/update_topic_freshness.py --tracker <temp.json>`
3. 看 cluster 結果：是否有「Agent 開發」和「AI 生成文案」被歸成一群？
4. 若有誤群 → 考慮調整 threshold / 加入停用詞 / 降低詞 token 權重

這步是 PoC，不產出正式程式碼。

### 還沒決定的事項

- **N 天視窗長度**（14 / 30 / 60 天？AK 預設是 30 天 `recent_day_window` + 10 篇 `recent_post_window`）
- **相似度呈現格式**（給使用者看的文字要怎麼寫？"最近 12 天前寫過相似內容（相似度 0.42）"？）
- **分數範圍與意義**（AK 是 0-100，我們沿用？還是自訂？）
- **要不要也檢查串文**（主貼文 vs 串文各篇的比對策略）
- **誤群風險驗證結果**（Q3 的 PoC 做完才能確定要不要調參）

### 下一步（按順序）

1. **PoC：誤群風險驗證**（Q3）— 用 AK 原 script 跑一次現有貼文，看繁中 AI/tech 會不會誤群
2. 根據 PoC 結果決定：直接採用 AK 的參數 / 調整閾值 / 加停用詞
3. 決定 N 天視窗、分數格式、呈現方式
4. 設計 `advisor.py` 整合介面（在哪呼叫、怎麼傳給 review prompt）
5. 實作與測試

## AI 生草稿工作流設計：C 路線（2026-04-22）

這一節是研究從「點到點吸收」擴展到「整體工作流架構」的結果。觸發點是挖出使用者既有的 `feat/advisor-plan` 分支歷史，和 4-14 那次實戰翻車。

- 2026-04-14 `planner.py` 串文骨架生成器實作完成（24 commits ahead of main）
- 實作：兩階段 AI 生骨架（Stage 1 選框架 → Stage 2 產串文每則鉤子/字數/內容方向/情緒）
- 資源：`references/copywriting-frameworks.md`（16+1 文案框架）、`drafts/*.plan.md` 範例
- PR #1 open，**未 merge**，原因見下方實戰診斷

### 4-14 實戰翻車診斷

Handoff：`docs/superpowers/handoffs/2026-04-14b-threads-advisor-step5-fail.md`

那晚嘗試用 skill 寫第一篇真實貼文（「從『我想要發文助手』到 skill 上線」），寫出草稿後使用者兩句評語：
> 「學得太過分了」
> 「一點都沒有創意」

使用者當晚自行歸納三個設計缺陷：

| 失敗類型 | 診斷 |
|---|---|
| **A. 語氣形式 cosplay** | `voice-patterns.md` 列「常用符號」被 AI 讀成「必撒清單」；機械堆疊、到字數就斷；形似神不似 |
| **B. 缺語氣錨點** | 三份 reference 給結構/哲學/語氣規則，但無實例；AI 只能從規則推，必然失真 |
| **C. 缺「選角度」步驟（核心）** | 草稿是記者式事件重述，不是分享者的思考過程；**工具沒強迫想清楚「這篇的銳利點」就開始寫** |

### 路線選擇：A / B / C

討論中把「AI 生草稿」細分三條路：

| 路線 | 描述 | 判定 |
|---|---|---|
| **A** | AI 從零寫、人微調 | ❌ 使用者品質標準是人工水位，短期達不到 |
| **B** | AI 跑流程（選題/框架/審查）、**內文全人寫** | 可行但保守 |
| **C** | AI 生骨架 + 人填內容 + AI 防味 | ✅ **選定** |

**校準**：AI 草稿常態是「還不錯、需要修」；4-14 是異常糟特例，不是通則。不要把單次翻車推到極端結論。

### 4-14 三診斷 × C 架構修復對照

| 4-14 失敗點 | C 架構修復層 |
|---|---|
| voice-patterns 被當「必撒清單」 | 第 4 層偵測 cosplay 堆疊（符號密度、機械斷行） |
| 缺「範例五」錨點 | 第 1 層 past-post 刺激（D 設計） + 第 3 層拉類似段落 |
| 缺「選角度」步驟 | 第 1 層整個就是這個 |

### 選角度 Gate 的設計候選（尚未選定）

| 設計 | 機制 | 優缺點 |
|---|---|---|
| **A. 固定問題清單** | AI 用一組固定問題追問 | 簡單；但久了有 AI 味 |
| **B. 模式配對** | 把「角度」分類成 meta-pattern（責任回歸型/角色翻轉型/市場心理型/時間錯位型），AI 套 patterns 產 1-3 候選 | 結構化；但可能重演 voice-patterns cosplay |
| **C. 蘇格拉底 Gate** | AI 不產角度、只問你問題逼你講出來；答得出來通過；答不出來告訴你「這題還沒準備好」 | 對應 handoff 診斷「Claude 不是給我答案，是給我問題」；但有些天會想繞過 |
| **D. 範例錨點對照** | 拉過去高表現貼文 → AI 抽角度特徵 → 對照新題目 | 用自己作品當 data；但需要足夠貼文量 |

**傾向方向**：C + D 組合。C 當主力（強制講得出銳利點），D 當後援（卡住時用過去貼文刺激）。

### 現況盤點

| 層 | 狀態 | 距離可用 |
|---|---|---|
| 1. 選角度 Gate | 完全沒有 | 從 0 設計，最需討論 |
| 2. 生骨架 | 實作完未 merge + 缺「角度輸入」升級 | merge + 小升級 ≈ 幾天 |
| 3. 填內文輔助 | 完全沒有 | 簡單功能，等 1 定了再做 |
| 4. AI 防味 | 完全沒有 | 借 AK + 客製 cosplay 檢測 ≈ 半天到一天 |

### 與既有候選點對照

- **第 4 層（AI 防味）= 原候選點 #2（AI-tone Detection）**，只是現在併入 C 路線的其中一環
- **第 1-3 層是新增**，原候選清單沒涵蓋
- **原候選點 #1（Semantic Freshness Gate）** 與 C 路線正交，可平行進行（它解「不要寫重複的」，C 解「不要寫沒角度的」）
- **原候選點 #3（Review → Tracker 迴路）** 仍是長期目標，與 C 路線無衝突

### 兩個設計問題（v6 已解，2026-04-23）

1. **「角度 Gate」願意被自己的工具拒絕嗎？** → **接受被拒絕，但以使用者決定為主**。soft gate：AI 會用對話表達擔憂，但使用者明確要寫就放行（記 override reason）。
2. **「選角度」這步要做成什麼形式？** → **互動式純 skill**（無代碼）。詳見下方「設計定稿（v6）」。

## 選角度 Gate 設計定稿（v6，2026-04-23）

從候選 A/B/C/D 出發，經 6 次迭代收斂。最終形態：**訪談者 + 共創者，雙向問答迴路收斂到共鳴**。

### 設計演進

| 版本 | 核心轉變 | 推翻原因 |
|---|---|---|
| v1 | 5 題固定 anchor + 強制對比句 | 重複 4-14 翻車——框架被當必撒清單 |
| v2 | 框架內化為 AI 探針，自然語言問 | 仍要使用者「想出銳利點」這個難題 |
| v3 | AI 從使用者素材萃取候選 | 缺學習維度 |
| v4 | + 思考過程透明化 + 反思機制 | 「教練」框架太說教 |
| v5 | 訪談者 + 共創者，AI 帶觀察進場 | 共鳴機制玄、AI 主導風險 |
| **v6** | **AI 永遠用問句出場，共鳴 = 問答迴路收斂** | **定稿** |

### 角色：Podcast 訪談者 + 共創者

不是教練、檢核員、寫作老師、SOP 流程。

氣質：真好奇、短評不長論、帶觀察但不堅持、願意現場改變想法。

### 流程：對話流動，無硬 stage

舊版線性切 stage 1 → 2 → 3 → 4。v6 改成對話流動，AI 自己感覺何時深聽 / 何時拋觀察 / 何時收網。唯一顯式時刻：最後鎖定切入點。

### 核心機制：AI 兩種發言形式（只有這兩種）

| 形式 | 用途 | 範例 |
|---|---|---|
| **發問** | 從素材或自己觀察拋出探針 | 「你提到 X 兩次，是不是這個對你特別重要？」 |
| **總結 + 詢問** | 階段性整理 → 求確認 | 「我聽下來覺得你想說的是 Y，你覺得呢？」 |

**禁止**：陳述句下定論（「切入點就是 X」）。

### 雙向萃取

AI 可以從使用者素材撈，**也能帶自己觀察進場**（廣泛知識 / 過去風格觀察 / 當下被刺激出來的想法）。

規則：AI 必須標明「這是我的觀察」vs「你的素材」，讓使用者自由接受或反駁。

### 共鳴的操作化（不再玄）

```
AI 觀察 → 包成問句拋出
   ↓
使用者回應（同意 / 修正 / 否定）
   ↓
AI 吸收 → 再萃取 → 再問
   ↓
反覆直到使用者「對，就是這個」
   ↓
= 共鳴 = 共識 = 鎖定切入點
```

共鳴不是 AI 內感，是迴路收斂到使用者點頭那一刻。可操作、可觀測。

### 內部透鏡（不對外）

AI 內部用四透鏡掃但不出現在對話：反直覺 / 對比張力 / 讀者收穫 / 自我重複。

只在使用者問「為什麼這算切入點」時才命名透鏡。平常不主動講。

### 學習維度收斂為隱性

v4 的「教練」收回，學習機制保留但隱性：
- 萃取時把線索講出來（自然透明）
- 反思機制預設**關**（使用者主動說「幫我反思一下」才開）
- 透鏡命名預設**關**（被動式，使用者問才講）

對話結構本身會讓使用者內化，不需顯式教學。

### Override

使用者明確說「不討論，直接幫我整理 angle 就好」→ AI 跳過討論，只做 v3 純萃取。

必須記錄 reason，未來可 review override 過的篇表現。

### 產出：`drafts/<slug>.angle.md`

```yaml
---
topic: 一句話標題
sharpness: 共創切入點（AI 整理，使用者點頭，不規定句型）
reader_value: 對讀者的價值
source: user_material | ai_observation | co_created
source_quotes: [從對話拉出支撐切入點的原話 1-2 句]
override: false
override_reason: ""
created_at: 2026-04-23
---

[完整對話紀錄]
```

`source` 三值意義：
- `user_material`：純從使用者素材萃取
- `ai_observation`：AI 帶進場觀察被使用者接受
- `co_created`：兩人問答跑出誰也沒料到的角度（**最珍貴**）

長期看 `co_created` 比例 = gate 健康度指標。

### 入口：純 skill（無代碼）

寫成 `threads-angle-gate` skill，零代碼，純 prompt engineering。等實測不穩再升級成 CLI + skill 包裝。

### 仍待決（實作前）

1. **Skill 細部 prompt**：問句範例、總結模板、引導語氣樣本
2. **與 planner.py 介接**：sharpness 怎麼傳給 planner.py 當主軸（環境變數 / argparse / 讀 `.angle.md`？）— 等第 2 層 planner 升級設計時一起決定
3. **過去貼文資料接入**（讓 AI 帶「使用者風格觀察」進場）：依賴候選點 #1 歷史資料層
4. **對話紀錄存檔策略**：完整對話可能很長，要不要 truncate？

## 第 2 層（planner.py 升級）設計定稿（2026-04-23）

> 接 v6 第 1 層產出的 `angle.md`，產出串文骨架。核心原則：**切入點是主軸方向，不是主旨**——讓表達在過程中自然圍繞它展開，不靠硬規則壓制。

### 設計哲學：從後面需要什麼，反推這一段需要什麼

第 2 層該吃什麼欄位，**不是看第 1 層產出什麼**，而是看第 3 層（填內文）跟最終 plan.md 產出需要什麼，反向推。

→ `angle.md` 保留所有欄位（題目、切入點、對讀者的價值、原話、override 狀態），下游可能會用到，刪了就沒彈性。

### 核心句（來自使用者）

> 切入點只是一個主軸跟方向，沒有主旨，但是要表達的東西就是在表達的過程，能夠圍繞它展開及表達，這樣就夠了。

這句話定下整層設計基調：**強引導，不強制**。

### 介接：讀整份 `angle.md`

| 場景 | 指令 |
|---|---|
| 主流程（走 gate） | `planner --angle drafts/<slug>.angle.md` |
| Fallback / Override | `planner --topic "X"` 跳過 gate，plan.md 開頭加軟警告 |

讀整份檔比拆欄位傳更有彈性，未來 angle.md 加新欄位下游不用改介面。

### Stage 1（選框架）改法

從「題目：X，挑 3 個框架」改成「題目：X，切入點：Y，挑能放大這個角度的 3 個框架」。

16+1 框架本身不動，挑選依據變得更精準（topic 決定大類、sharpness 決定具體框架）。

### Stage 2（生骨架）改法：強引導 + 軟提醒

- 切入點放 prompt **最顯眼位置**（題目下方第二行）
- 寫「這篇的核心切入點是 ____，骨架圍繞這個展開」
- **不寫**「每則必須扣回切入點」（會壓死鋪陳 / 反轉 / 對比這些好結構）
- 軟提醒：「如果某則跟切入點關係不明顯，想清楚為什麼要保留」

### 為什麼不強制每則扣回？

| 不強制的理由 | 說明 |
|---|---|
| 對治流水帳的根因不是規則 | 4-14 翻車根因是「沒有切入點」。第 1 層 Gate 已經解了這個。根因解了就不需要硬規則。 |
| 強制會壓死好結構 | 鋪陳型（前 2 則鋪情境→第 3 則出切入點）、反轉型（前面假裝 A→最後翻 B）、對比型，都不是「每則扣回」 |
| 跟 v6 精神矛盾 | v6 整個就是「不要把框架當必撒清單」。強制每則扣回也是必撒清單的另一種形式 |
| 強引導已經夠 | 切入點在 prompt 最前面 + 上下文夠密 → AI 自然圍繞，不需要硬規則 |

### 第 2 層 prompt 主動用三個欄位

`angle.md` 保留全欄位，但第 2 層 prompt 主動用：
- **題目**：產出 plan.md 的標題
- **切入點**：Stage 1 選框架依據 + Stage 2 骨架主軸
- **對讀者的價值**：填入 plan.md「目標受眾 → 痛點 / 動機」段

**原話**：當「上下文」放在 prompt 後段，AI 自己決定要不要參考。不主動用，但保留可用性。

### 不新增階段

Stage 1 + Stage 2 已經夠。第 1 層 Gate 把「想清楚」做完了，planner 不該再多想，只該照 sharpness 展開。多階段會讓簡單的事複雜化。

### Override 警告：軟版本

跳過 gate（直接用 `--topic`）→ plan.md 開頭加：

```
⚠ 警告：本骨架未經角度 gate，可能缺少主軸。建議產出後重 review。
```

**硬版本**（依賴候選點 #3 的數據，例：「過去跳過 gate 的篇互動率平均低 X%」）等候選點 #3 落地才做。

### 仍待決（實作前）

1. **angle.md 路徑慣例**：`drafts/<slug>.angle.md` vs 其他位置？
2. **完整對話紀錄存檔策略**：對話紀錄完整附在 angle.md 末尾，還是只放 sharpness 出處的原話？跟第 1 層待決第 4 點綁定。
3. **Fallback 觸發**：CLI 旗標明確切換，還是「沒 angle.md 就自動 fallback」？

## 第 4 層（AI 防味）設計定稿（2026-04-23）

> 發文前掃描，標記 AI 味重的地方。從 AK 借核心通用項 + 客製 cosplay 堆疊檢測，接進 advisor review 第 7 維度。**全程提供建議，不擋住發文**。

### 設計哲學

延續第 2 層「強引導，不強制」：
- 掃描結果是**參考資料**，不是強制規則
- 嚴重度報告呈現給使用者，由使用者決定要不要改

### 核心句（來自使用者）

> AK 那套可以參考，但是不一定要非得應用。

這句話定下整層基調：**參考性**先於套用性。

### 三步驟：借 + 加 + 整合

**1. 借（從 AK 借核心通用項，跳過領域特定項）**

AK 給了 20 個檢測項（10 語句層 + 5 結構層 + 5 內容層）+ 7 個降 AI 味方法。

| 借 | 跳 |
|---|---|
| 哲理收尾、表演式轉折、書面連接詞、金句密度、判斷太完整、對比句工整、反問鎖結論、情緒詞不自然、列舉太工整、論證太順、資訊分布太均勻、抽象判斷缺案例、觀點太中立 | 「固定句式清單」（AK 是 SEO 圈用語，「說白了就是」這類我們不一定適用）等領域特定項 |

具體借哪些等實作時再篩。原則：**參考但不強制套用**。

**2. 加（客製 cosplay 堆疊檢測，對治 4-14 翻車）**

AK 沒涵蓋「voice-patterns 被當必撒清單」這種失敗模式，要自己加：

| 項目 | 觸發條件 |
|---|---|
| 符號密度過高 | 每段符號數（破折號、引號、刪節號）超過平均 1.5 倍 |
| 機械斷行 | 句子在語意未完處斷行 + 各則字數標準差 < 5 字 |
| voice-patterns 過度命中 | 單篇命中 voice-patterns.md 清單項目 ≥ N 個（反向檢測「你抄了規則的痕跡」） |

第 3 項最關鍵——我們自己有 `voice-patterns.md`，可以拿來反向檢測。

**3. 整合（接進 advisor review 第 7 維度）**

兩階段：

```
草稿
  ↓
結構化掃描（純 Python / regex / 計數）
  → 產出檢測報告（命中項目、嚴重度）
  ↓
報告餵給 advisor review 當 input
  ↓
Codex 在第 7 維度整合判斷
  → 哪些是真問題 / 哪些 false positive / 修改方向
```

結構化掃描給可重複的**確定性**，AI 整合給**判斷**過濾 false positive。

### 嚴重度分級（沿用 AK）

**單項觸發**：
- 確定（pattern 在人類寫作中幾乎不出現）
- 可能（單獨出現不是問題，多項疊加才加重）

**全篇總評**：
- 低（0-2 個確定項）：基本沒問題
- 中（3-5 個確定項）：建議針對性修改
- 高（6+ 個確定項）：建議大幅改寫

**不擋住發文。** 高等級只是強烈建議改，不會 block。跟第 2 層 override 哲學一致。

### 報告格式

```
## AI 味檢測報告

### 確定有 AI 味（建議修改）
- [行/段] 具體哪句 → 命中規則 → 修改方向

### 可能有 AI 味（自行判斷）
- [行/段] 具體哪句 → 命中規則 → 為什麼標記

### 結構/內容層問題
- [描述] → 問題 → 修改方向

### AI 味濃度評估
- 觸發項：X 個（確定 Y / 可能 Z）
- 等級：低 / 中 / 高
```

### 仍待決（實作前）

1. **AK 借的具體清單**：實作時從 20 項裡篩出最終採用清單
2. **voice-patterns 命中閾值 N**：3 個？5 個？要實測校準
3. **報告呈現位置**：跟 advisor review 報告同檔，還是獨立 detection 報告？
4. **結構化掃描實作方式**：純 Python regex / 借 AK 原 script / LLM 自掃？

### 與其他層的接點

- **與第 2 層 plan.md**：第 2 層產骨架，使用者填內文後跑第 4 層
- **與 advisor review**：第 4 層成為 advisor review 第 7 維度
- **與第 1 層 angle.md**：無直接接點（第 1 層管角度，第 4 層管表達）

## 第 3 層（填內文輔助）設計定稿（2026-04-23）

> 接第 2 層 plan.md 骨架後、第 4 層防味前。**核心：人寫內文，工具輔助**——不是 AI 生成內文。

### 設計哲學

C 路線最薄的一層。創作核心是人的事。工具的角色是**陪伴**，不是**代替**。

### 兩個小幫手（其他都不做）

**幫手 1：顯示當前 P 的骨架 metadata**

開始填某則 P 時，AI 主動把骨架資訊放出來：
- 這則要講什麼（內容方向）
- 鉤子類型
- 建議字數
- 情緒

防止使用者寫到一半忘記骨架。

例：
> 「骨架說 P3 要講『AI 不是替你做，是逼你想清楚』，鉤子是反問句，字數約 300。要開始嗎？」

**幫手 2：手動拉歷史類似段落當參考**

使用者卡住時，主動跟 AI 說：「過去寫類似的怎麼寫？」AI 從 SQLite 撈相似段落。

- **不是抄**，是看自己過去風格找感覺
- 跟候選點 #1（Freshness Gate）共用相似度演算法
- 兩個用途不同：Freshness 警告重複，第 3 層給靈感

### 明確不做的東西

| 不做的 | 原因 |
|---|---|
| 字數即時提醒 | 干擾，無意義 |
| AI 自動修句子 | 違反「人寫，工具不碰創作核心」 |
| 邊寫邊掃 AI 味 | 打斷思路。寫完才掃 = 第 4 層工作 |
| 工具自動撈歷史段落（不問就丟） | 打斷思路 |

### 形式：純 skill（Agent 對話）

跟第 1 層 Gate 一致：使用者「不打 CLI」、Agent 對話為主。零代碼。

可以是 `threads-fill` skill，或併入 `threads-angle-gate` skill 當下一階段。

### 第 3 層的設計特性

- **最薄一層**：應該最薄，創作核心是人的事
- **與候選點 #1 共用底層**：相似度演算法兩邊都用，可以一起設計
- **無新模型 / 新 prompt**：不需要 LLM 生成東西，只需要呈現現有資料 + 撈相似段落

### 仍待決（實作前）

1. **要併入 `threads-angle-gate` skill 還是獨立 `threads-fill` skill？** 併入更連貫，獨立更模組
2. **撈歷史段落的相似度閾值**：跟候選點 #1 PoC 結果決定
3. **歷史段落呈現格式**：完整貼文 vs 只摘相似段落？

### 與其他層的接點

- **與第 2 層 plan.md**：讀骨架的 metadata 顯示
- **與第 4 層**：第 3 層結束（人寫完）→ 第 4 層接力（掃 AI 味）
- **與候選點 #1**：共用 SQLite 相似度演算法
