# 資料、格式、與程式的本質

**日期**：2026-04-15
**脈絡**：在討論「CLI 對 App 開發的幫助（角度 A）」時，從「dict 是什麼」一路推導出來的心智模型。記下來當作未來學新技術的參考地圖。

---

## 1. 資料 vs 格式是兩件事

同一份資訊，在不同場景會穿不同衣服：

| 名稱 | 在哪裡用 | 長相 |
|---|---|---|
| **dict** | Python 程式執行時，記憶體裡 | `{"id": "123", "likes": 42}` |
| **物件** | Python 程式裡，dict 的「正式版」 | `Post(id="123", likes=42)` → 用 `.id` 取 |
| **JSON** | 傳輸 / 儲存時的文字格式 | `'{"id": "123", "likes": 42}'`（是字串） |
| **YAML** | 設定檔（人類友善） | `config.yaml` |
| **.env** | 環境變數 | `THREADS_ACCESS_TOKEN=xxx` |
| **資料庫表格** | 永久儲存、查詢 | SQLite / Postgres |
| **Markdown** | 給人讀的報告 | `report.md` |

**規律**：
- 程式內部運算 → dict / 物件
- 要跨出去（存檔、傳 API、回前端） → 轉 JSON
- 別人傳 JSON 進來 → 轉回 dict 才能處理
- 本機設定 → YAML / .env
- 永久存 → 資料庫

---

## 2. 流程示意

```
[手機 App]  ←─── JSON ───→  [伺服器]  ←── dict ──→  [資料庫]
                (傳輸)         (處理)                 (儲存)
```

每個階段換一次形式，**但內容是同一批貨**。

類比：
- 工廠內部：輸送帶上的東西（dict，方便加工）
- 要寄出去：裝箱貼標（JSON，方便運輸）
- 倉庫存放：上架分類（資料庫）

---

## 3. 程式的本質 = 資訊的收集、處理、分發、運用

不管介面長什麼樣（CLI / Web / Mobile / Bot），骨子裡都是：

```
輸入（資訊） → 處理 → 輸出（資訊）
```

學術上叫 **Everything is data**（萬物皆資料）或 **Data abstraction**（資料抽象）。

---

## 4. 所有東西都是資訊

電腦看到的世界全是數字：

| 人看到的 | 電腦看到的 |
|---|---|
| 一張照片 | RGB 數字陣列：`[[255,0,0], [128,64,32], ...]` |
| 一首歌 | 聲波取樣值：`[0.02, -0.05, 0.1, ...]` |
| 一部影片 | 很多張照片 + 一串聲音 |
| 一篇貼文 | UTF-8 字元編碼：`[228, 184, 173, ...]` |
| 一個按讚 | 一個 bit：`1` |
| **程式碼本身** | 文字字串（所以 AI 才能讀寫它） |

**全部都是數字陣列**。處理圖片的 OpenCV 和處理文字的 json 模組，底層都是讀數字、算數字、寫數字。

---

## 5. 這個觀念的實用價值

### 學新技術時的通用問法

遇到任何新技術（AI、向量資料庫、GraphQL、MongoDB…），問自己：

1. **輸入是什麼資訊？**
2. **輸出是什麼資訊？**
3. **中間怎麼處理？**
4. **用什麼格式存 / 傳？**

例：AI 熱詞解密
- **embedding** = 把文字變成一串數字（換個格式）
- **向量資料庫** = 專門存這些數字的倉庫
- **RAG** = 查倉庫找相關數字 → 丟給 AI → AI 吐新數字 → 變回文字

還是資訊的收集、處理、分發。只是資訊形式變成高維數字向量。

### 架構可以跨題材遷移

`threads_pipeline` 現在的骨架（fetch → analyze → store → report）能直接延伸到：

| 產品 | 資訊流程（架構相同） |
|---|---|
| IG 貼文分析 | 收圖片+文字 → AI 分析 → 輸出建議 |
| YouTube 選題助手 | 收熱門影片標題 → AI 分析 → 輸出選題 |
| Podcast 摘要 | 收音檔 → 轉文字 → AI 摘要 → 輸出重點 |
| 短影音剪輯助手 | 收影片 → 分析節奏 → 輸出剪輯建議 |

換題材只是換中間的處理函式，流程骨架不動。

---

## 6. 對應到現在的 threads_pipeline

同一個專案裡同時用了六種格式：

```
.env              → 環境變數（存 Token）
config.yaml       → YAML（設定檔，人看）
Threads API 回應  → JSON（跨系統傳輸）
程式裡的 data     → dict（處理運算）
SQLite            → 表格（永久儲存）
report.md         → Markdown（給人讀）
```

**六種格式，同一批資訊**，在不同階段換衣服。

---

## 7. 一句話帶走

> **JSON 是寄出去的明信片，dict 是拆開放在桌上看的那張紙。內容一樣，只是狀態不同。**

> **程式 = 資訊處理。** 換題材就是換處理函式；換介面就是換輸入輸出的格式。核心邏輯（服務層）應該獨立於介面存在。
